# Smart Band Sync v1 Envelope

Status: Q6-1 envelope core, version 1.0. This document freezes only a
transport-independent frame envelope. Payload bytes remain opaque.

## Byte layout

All multi-byte integers are unsigned and little-endian. Implementations must
encode fields individually; a C struct is not a wire representation.

| Offset | Width | Field | v1.0 rule |
| ---: | ---: | --- | --- |
| 0 | 2 | magic | `0x4253`, encoded as ASCII `53 42` (`SB`) |
| 2 | 1 | protocol major | `1` |
| 3 | 1 | protocol minor | `0` |
| 4 | 1 | frame type | `1..4` as listed below |
| 5 | 1 | flags | bits 0 and 1 defined; bits 2..7 reserved as zero |
| 6 | 1 | status | `0..4` as listed below |
| 7 | 1 | reserved | must be zero |
| 8 | 2 | payload length | `0..224`, excluding header and CRC |
| 10 | 4 | transaction ID | correlation value; lifecycle belongs to the service |
| 14 | 2 | sequence | envelope value; ordering policy belongs to the service |
| 16 | 2 | chunk index | envelope value; reassembly belongs to the service |
| 18 | N | payload | `N = payload length`; opaque bytes |
| 18 + N | 2 | CRC16 | little-endian CRC over bytes `[0, 18 + N)` |

The fixed header is 18 bytes. The CRC trailer is 2 bytes. The payload maximum
is 224 bytes, so the maximum complete frame is 244 bytes. This bound fits in the
application value of a common 247-byte BLE ATT MTU, but v1 does not require that
MTU and does not implement MTU discovery or fragmentation.

## Enumerated fields

Frame types are `DATA=1`, `CONTROL=2`, `ACK=3`, and `ERROR=4`. The codec checks
only that the value is in this set. It does not implement ACK handling or infer
payload schemas from the type.

Flag bit 0 is `MORE_CHUNKS`; flag bit 1 is `ACK_REQUIRED`. Carrying a flag does
not implement chunk reassembly, acknowledgement, timeout, or retry behavior.

Status values are `OK=0`, `INVALID_REQUEST=1`, `UNSUPPORTED=2`, `BUSY=3`, and
`INTERNAL_ERROR=4`. The codec validates the numeric value but does not enforce
service-level type/status combinations.

## CRC16

The checksum is CRC-16/CCITT-FALSE:

| Parameter | Value |
| --- | --- |
| Width | 16 |
| Polynomial | `0x1021` |
| Init | `0xffff` |
| RefIn / RefOut | false / false |
| XorOut | `0x0000` |
| Check input | ASCII `123456789` |
| Check value | `0x29b1` |

Coverage is the complete fixed header followed by the payload. The two-byte CRC
trailer is excluded. The numeric CRC is serialized little-endian.

## Version and length rules

The decoder accepts protocol major `1` and minor values less than or equal to
its supported minor. The v1.0 decoder therefore accepts only `1.0`; an unknown
major or future minor returns `BAD_VERSION`. A future compatible minor can be
accepted only after the implementation raises its supported minor and confirms
that all newly defined bits remain safely understood.

Decode requires the input size to equal exactly `18 + payload_length + 2`.
Shorter data returns `TRUNCATED`; longer data returns `TRAILING_DATA`. A declared
payload over 224 returns `BOUNDS` before any payload access. All reserved bits
and the reserved byte must be zero.

## API and aliasing

Encoding validates the complete request and output capacity before writing.
The payload source bytes may overlap the output frame bytes, including an
in-place source beginning at `output`; the implementation uses `memmove` before
writing the header. The test suite freezes this supported aliasing behavior.
The frame descriptor and `encoded_size` objects are ordinary metadata objects
and must not be placed inside either byte range.

Decoding is zero-copy. On success, `view.payload` points at byte 18 of the input
frame, so the caller must keep the complete input frame unchanged for the life
of the view. The view object itself must not overlap the input frame. On every
decode failure the view is cleared.

## Decode and encode results

| Result | Meaning |
| --- | --- |
| `OK` | Complete valid frame |
| `INVALID_ARGUMENT` | Null required pointer or null non-empty payload |
| `BUFFER_TOO_SMALL` | Encode destination cannot hold the required frame |
| `BOUNDS` | Payload length exceeds 224 |
| `TRUNCATED` | Fewer bytes than the declared exact frame size |
| `TRAILING_DATA` | More bytes than the declared exact frame size |
| `BAD_MAGIC` | Magic is not `SB` |
| `BAD_VERSION` | Unsupported major or future minor |
| `BAD_TYPE` | Frame type is outside `1..4` |
| `BAD_FLAGS` | A reserved flag bit is nonzero |
| `BAD_STATUS` | Status is outside `0..4` |
| `BAD_RESERVED` | Reserved byte is nonzero |
| `BAD_CRC` | Stored and calculated CRC16 differ |

## Independent golden vector

The example frame is stored in
`tests/vectors/sync-v1-envelope.json`. Its header and payload were assembled
from the table above. The expected CRC was independently checked with Python's
`binascii.crc_hqx(data, 0xffff)`, rather than generated by the production C
implementation.

```text
53 42 01 00 01 03 00 00 04 00 78 56 34 12 01 02 03 04
de ad be ef c8 a6
```

## Deliberately not frozen

History, settings, notifications, and all other business payload schemas remain
unversioned opaque bytes at this layer. Duplicate detection, ordering,
transaction state, ACK policy, retry policy, timeout, chunk reassembly,
pairing, authentication, encryption, GATT, and BLE transport behavior belong to
later sync-service and transport work.

The deterministic malformed-frame test is a repeatable robustness regression,
not a security fuzzing result or a memory-safety proof.
